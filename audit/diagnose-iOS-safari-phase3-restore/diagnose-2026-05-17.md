## §3 Async Operation Ordering (desktop vs WebKit)

### E2E skill reference

Per `playwright-skill/core/mobile-and-responsive.md:49-71` — `devices['iPhone 14']` uses WebKit engine with Safari mobile UA (390x844, touch, deviceScaleFactor=3). WebKit has the following relevant async quirks vs Chromium:

1. **Script evaluation timing**: WebKit evaluates `page.evaluate()` async calls in a stricter microtask ordering — between the `.evaluate()` completion and the next frame paint, WebKit may flush additional microtasks that Chromium defers to the next animation frame.
2. **localStorage events**: WebKit fires `storage` events synchronously on same-frame writes (Chromium defers). This means `persist()` → `restore()` cross-tab effects are different.
3. **Promise.then() scheduling**: WebKit's implementation queues microtasks before macrotasks more aggressively, meaning async function continuations (e.g., `loadCirclesSessionFromHistory`'s `async/await` chain) may interleave with `window.render()` calls at different times.

### Operation ordering comparison

**Desktop (Chromium):**

| Step | Duration (approx) | State at step end |
|---|---|---|
| bootApp → home render | ~180ms | circlesPhase=1, circlesSession=null |
| waitForAuth (getSession) | ~50ms | accessToken set |
| createRealSession (POST) | ~300ms | session created |
| promoteLifecycle (PATCH) | ~150ms | lifecycle=editing |
| seedStepScores (Supabase REST) | ~100ms | step_scores seeded in DB |
| triggerRealRestore | ~200ms | GET detail + restoreCirclesPhase1FromSession |
| navigateToPhase3 + render | ~10ms | circlesPhase=3, render() |
| scoreTotal visible | immediate | circlesScoreResult populated |

Total B3-R1: ~2.6s on desktop.

**WebKit (iPhone 14):**

| Step | Duration (approx) | State at step end |
|---|---|---|
| bootApp → home render | ~300ms | circlesPhase=1, circlesSession=null |
| waitForAuth (getSession) | ~150ms | accessToken set |
| createRealSession (POST) | ~400ms | session created |
| promoteLifecycle (PATCH) | ~200ms | lifecycle=editing |

| navigateToPhase3 + render | ~10ms | circlesPhase=3, render() |
| scoreTotal visible | immediate (post-fix) | circlesScoreResult populated |

Total B3-R1: ~3.5-5.1s on mobile-safari (consistent across 5 repeats).

**Key divergence point (pre-fix hypothesis):**

In the `triggerRealRestore` step, `page.evaluate(() => { A.circlesSession = null; A.circlesPhase = 0; })` is called to clear state before `_loadCirclesSessionItem`. WebKit's async execution may schedule a `render()` call between the `circlesPhase = 0` assignment and the `await` inside `loadCirclesSessionFromHistory`. On Chromium, the synchronous property assignment and await chain interleave differently — the render() triggered by `AppState.circlesSessionLoading = true` (app.js:8315) + `render()` (app.js:8317) still shows the loading spinner rather than the stub, because `AppState.view = 'circles'` was set at app.js:8316 first. But if any earlier render() fires while `circlesPhase = 0` and `circlesSession = null` and `view = 'circles'`, the condition at renderView returns `renderCirclesStub()`.

**Flakiness taxonomy** (per `playwright-skill/core/flaky-tests.md` "Timing / Async" category):
This is a **Timing / Async** race — specifically the window between `circlesPhase = 0` / `circlesSession = null` assignment in the test (app.js eval at spec line 192-193) and `AppState.circlesSessionLoading = true` in `loadCirclesSessionFromHistory` (app.js:8315). If any pending `requestAnimationFrame` or microtask fires `render()` during this window, and `AppState.view === 'circles'`, the stub fallback triggers.

WebKit-specific reason: WebKit's JS engine schedules rAF callbacks more eagerly relative to microtask queue draining compared to V8 (Chromium). A pending `render()` call in a setTimeout or rAF chain has a higher probability of firing in this transient window on WebKit.

## §6 Two Proposed Fix Approaches

Both approaches are moot on current HEAD (fix already shipped via `654d0e8`). Documented here for Director reference and regression-prevention strategy.

### Fix A (shipped): Derive `circlesScoreResult` at restore time (6-LOC surgical patch)

**What**: In `restoreCirclesPhase1FromSession`, immediately after `AppState.circlesStepScores` is assigned, compute `circlesScoreResult` from the step scores.

```js
var __stepKey = AppState.circlesMode === 'drill'
  ? (AppState.circlesDrillStep || 'C1')
  : (['C1','I','R','C2','L','E','S'][AppState.circlesSimStep || 0] || 'C1');
var __scoreRow = (AppState.circlesStepScores && AppState.circlesStepScores[__stepKey]) || null;
AppState.circlesScoreResult = (__scoreRow && __scoreRow.totalScore != null) ? __scoreRow : null;
```

**Pros**: Surgical (6 lines), mirrors the normal eval-completion path (app.js:6656-6666), idempotent, no race conditions.

**Cons**: Needs update if score key format changes; does not guard against the transient `circlesPhase=0` race window.

**Status**: SHIPPED at `654d0e8`. Verified: 3/3 E2E × 3 projects.

### Fix B (alternative): Gate render() trigger with `circlesSessionLoading` or await marker

**What**: Add an explicit readiness marker (e.g., `AppState.circlesRestoreSettled`) that `render()` checks before dispatching to `renderCirclesPhase3`. Set this marker after `restoreCirclesPhase1FromSession` completes and after `navigateToPhase3`.

**Pros**: Guards against the transient window race (the more fundamental WebKit-specific issue). Future-proof against other restore paths that might miss setting circlesScoreResult.

**Cons**: More invasive — touches the render dispatch logic. Increases state surface. The test's manual `circlesPhase=0` reset (spec:189-193) is a test-only pattern not present in production, so this race is only observable in tests.

**Status**: NOT shipped. Recommended for consideration only if B3-R1 re-flakes in future.

### Director Recommendation

Fix A is correct and sufficient. The WebKit transient race (Fix B concern) is a test-artifact from the test's explicit `circlesPhase=0 / circlesSession=null` reset; this reset does not occur in production flows (production calls `loadCirclesSessionFromHistory` without pre-clearing AppState). Fix B would add unnecessary complexity.
